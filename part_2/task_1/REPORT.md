# Отчёт: сравнение моделей семантической сегментации (часть 2)

## Постановка задачи

Семантическая сегментация облаков точек на датасете промышленных клапанов (PLY, поле `scalar_Label`, 10 классов).  
Для каждого сэмпла берётся **4096** точек; train/val/test — **70% / 15% / 15%**, seed **42**.  
Метрики на тесте: **OA** (overall accuracy), **mIoU**, **macro-F1**, IoU по классам 0–9.

Скрипты: `part_2/task_1/scripts/train_*.py`. Сырые метрики: `part_2/task_1/task_*/metrics/results.csv`.

**DGCNN** заметно слабее, но остаётся рабочим baseline.  
**KPConv** в текущей конфигурации не сошёлся: метрики на уровне случайного угадывания для ряда классов.

### Наблюдения

1. **Сложный класс 4** — у всех успешных моделей самый низкий IoU (~0.67–0.69). Вероятно, мало примеров или сильное пересечение с соседними классами.
2. **DGCNN** проседает на классах 1, 2, 4 (IoU &lt; 0.5 для 2 и 4).
3. **Point Transformer** выигрывает за счёт классов 2 и 7 относительно PointNet++.
4. **KPConv**: почти нулевой IoU на классах 2–4 — признак недообучения или несоответствия упрощённой реализации KPConv масштабу/плотности данных (нужна отладка grid size, числа kernel points, аугментаций).

## Архитектуры (кратко)

| Модель | Идея | Плюсы на задаче | Минусы |
|--------|------|-----------------|--------|
| **PointNet++** | Иерархический сэмплинг + локальные MLP | Стабильный результат, быстрее сходится (50 эпох) | Слабее на мелких/редких классах |
| **DGCNN** | Граф соседей в feature-space | Простая реализация, мало параметров на граф | Хуже обобщение на 10 классах |
| **Point Transformer** | Self-attention по точкам | Лучший mIoU/F1, сильные классы 2 и 7 | Дольше обучение (100 эпох) |
| **KPConv** | Свертки на kernel points в 3D | SOTA на крупных сценах при правильном препроцессинге | Здесь — плохая сходимость без тюнинга |

## Выводы

1. Для данного датасета клапанов **оптимальный выбор — Point Transformer**, с **PointNet++** как практичной альтернативой (почти тот же OA, чуть ниже mIoU).
2. **DGCNN** годится для сравнения baseline, но для сдачи/продакшена на этом наборе слабоват.
3. **KPConv** требует доработки (гиперпараметры K, NUM_KP, subsampling `NPOINTS`, баланс классов, больше эпох или learning rate schedule) перед повторным сравнением.
4. Общий узкий класс — **#4**; имеет смысл проверить распределение меток и class weights во всех скриптах.
